# Lessons — flutter_inactive_timer 실증

이 repo 가 `theflow` 의 각 단계에서 실제로 **무엇을 놓쳤나** 의 기록 — 규칙에 무게를 주는
근거. 전부 이 repo 에서 실제로 일어났다. 단계 번호는 `theflow` SKILL.md 와 일치. 바인딩
(`theflow.md`)이 추상으로 읽히면 여기 사건과 대조하라.

---

## 반복 실패: `await` 뒤에는 세상이 이미 바뀌어 있다

이 패키지는 **비동기 네이티브 읽기 위의 타이머**다. `getIdleDuration()` 을 `await` 하는
사이 사용자가 `stopMonitoring()`·`continueSession()`·`dispose()` 를 부를 수 있다. **같은
모양의 버그가 네 번** 났다:

- **`07ac425`** — `_checkInactivity` 가 await 뒤 `isMonitoring` 을 다시 안 봐서 stop 이후에
  콜백이 발화했다.
- **`9759514`** — example 앱이 await 뒤 `BuildContext` 를 그대로 썼다.
- **`8c03f6f` (#5)** — await 뒤 타이머를 arm 해서 선점된 스케줄이 유령 타이머를 남길 수
  있었다. **generation 카운터**로 봉했다.
- **`a6abd0f`** — `remaining()` 이 idle 읽기에 park 된 사이 `stopMonitoring()` 이 끼어들었다.

→ **await 를 새로 넣거나 옮겼으면 재개 지점에서 읽는 모든 상태**(`isMonitoring`,
generation, `mounted`)를 다시 확인하고, 그 경쟁을 재현하는 테스트를 붙인다.

## Step 1 — 이슈 먼저 (본문 근거도 실증 대상)

- **#5 (근거가 이미 거짓)**: 본문은 `_scheduleNextCheck()` 가 두 `Timer` 를 만들어 유령
  타이머가 샌다고 단언했다. 착수해 보니 그 누수는 **이미 #3 의 `InactivityPolicy` 추출로
  사라져 있었다** — `_arm` 이 재할당 전에 cancel 하고 `_pump` 가 await 뒤 상태를 다시 읽는다.
  실제 변경은 "누수 수정" 이 아니라 generation 카운터를 통한 **불변식의 명시화**가 됐고,
  커밋 메시지에 정정을 남겼다.

## Step 2 — 추측 금지 (네이티브·하네스도 실측)

- **ADR-0001 (네이티브 소스를 직접 읽는다)**: Windows 의 `getSystemTickCount` 는 64비트
  `GetTickCount64` 에서, `getLastInputTime` 은 **32비트** `GetLastInputInfo.dwTime` 에서
  왔다. 후자는 약 **49.7일**마다 wrap 하고, 그 뒤 Dart 뺄셈은 쓰레기값을 낸다. macOS 는
  마침 둘 다 `systemUptime` 기준이라 계약이 암묵적·비강제로 굴러가고 있었다. **"두 클럭을
  Dart 에서 뺀다" 는 계약 자체를 없애고** 네이티브가 idle duration 하나를 반환하게 했다.
- **#17 (테스트 하네스도 검증 대상)**: 새 테스트가 계속 실패해 대상 코드를 의심했는데 범인은
  fake 였다 — `GatedIdlePlatform.releaseHeldRead` 가 읽기가 park 되는 순간 `_hold` 를 null
  로 만들어 park 된 read 를 **release 할 방법이 없었다.** `_held` 로 따로 추적해 고쳤다.
  기존 유령 타이머 테스트는 generation 가드가 재개된 읽기를 버려 이 버그를 못 드러냈다.

## Step 3 — 설계 판단 코드 전에 (ADR 세 개가 코드보다 먼저)

- **ADR-0002 (표현 가능한 불법 상태를 없앤다)**: "`notificationPer` 를 deprecated 로 한
  릴리스 병존" 을 기각한 건 코드 전이었다 — 두 방식이 공존하면 생성자에서 "둘 다 세팅됨"
  이라는 불법 상태가 되살아나, sealed `NotificationTrigger` 를 도입한 이유를 무너뜨린다.
- **ADR-0003 (시그니처는 red 전에)**: `remaining()` 이 pull/push·sync/async 인지는 red
  전에 정해야 했다. 동기 getter 는 fresh idle 을 못 읽어 Notification 이전 구간의 사용자
  활동을 반영 못 한다 — 그 사실이 시그니처를 `Future<Duration>` 으로 확정했다.

## Step 4 — 증명 방법 (측정 대상이 측정 중에 움직인다)

- **#19 (유휴 시간 parity 테스트가 "입력 없음"을 가정하고 있었다)**: FFI 값과 채널 값을
  한 배치로 읽고 차이가 허용오차 안인지 보는 테스트였는데, 허용오차를 **"입력이 없으면
  유휴 시간은 늘기만 한다"** 는 전제 위에 세웠다. Windows 실기 첫 실행에서
  `{method channel: 47, windows/GetLastInputInfo: 0}` 으로 터졌다 — FFI 가 **더 작았다.**
  두 읽기 **사이에 실제 입력이 들어온** 것(앱 창 생성 시점의 입력 이벤트)이고, 리더 불일치가
  아니었다. 그대로 뒀으면 CI 에서 상시 flaky.
  → 두 상황은 **부호로 구분된다**: 나중 읽기가 **더 크면** 리더 불일치(잡아야 할 버그),
  **더 작으면** 배치 중 입력(무효 표본). 후자는 실패가 아니라 **폐기 후 재시도**로 바꾸고,
  재시도 상한과 폐기 건수 로그를 붙였다 — 상한이 없으면 "항상 낮게 읽는 소스"가 영원히
  재시도한다. **환경을 공유하는 값을 비교하는 테스트는, 비교 도중 환경이 바뀌는 경우를
  실패가 아니라 무효로 처리해야 한다.**

- **#23 (무효 판정은 부호가 아니라 정합성으로 해야 한다)**: 위 규칙을 "유휴 시간이 경과
  시간만큼 증가한다" 테스트에 옮기면서 `growth < 0` 만 폐기했더니, `idle grew by 1ms
  while 605ms passed` 로 실패했다. 입력이 창 **중간**에 들어오면 증가량이 **작은 양수**가
  된다 — 99ms 에서 리셋되고 끝에 100ms 면 +1. 음수가 아니라 가드를 통과했다.
  → 올바른 판별자는 부호가 아니라 **설명의 정합성**이다: 작은 증가량을 리셋으로 설명하려면
  **끝 값이 이 창 안에서 시작했을 만큼 작아야** 한다. 작은 증가량 + 큰 값은 리셋으로
  설명이 안 되므로 그건 **바인딩이 틀린 것**이고 실패해야 한다. **"무효 표본"과 "진짜
  버그"를 가르는 조건은 관측값 하나가 아니라 관측값들이 서로 앞뒤가 맞는지다.**

- **#23 (읽기 자체가 시간을 먹는다 — 허용오차가 아니라 비교 대상이 틀렸던 경우)**:
  "유휴 시간이 경과 시간만큼 증가한다" 를 `growth ≈ stopwatch(전체)` 로 단언했더니 macOS
  에서 `grew by 607ms while 612ms passed` 로 실패했다. 스톱워치는 `read1 시작 → 대기 →
  read2 끝` 을 재는데 growth 는 두 **표본 시점** 사이 간격이라, read1 이 값을 뜬 뒤와
  read2 가 값을 뜨기 전 시간만큼 항상 짧다. IOKit 워크는 사전을 통째로 만드는 무거운
  호출이라 그게 수 ms.
  → **Windows 는 이걸 못 잡았다** — 눈금 허용치 16ms 가 읽기 비용을 덮었다. 나노초 해상도라
  허용치가 2ms 인 macOS 가 비로소 5ms 를 노출시켰다. 허용오차를 넓히는 건 오답이고,
  참값이 두 시각 **사이**에 있으니 **상한(바깥 스톱워치)과 하한(안쪽 스톱워치) 을 둘 다
  재서 구간으로 가두면** 읽기 비용 상수가 아예 필요 없어진다.
  **관대한 허용오차는 통과시키는 게 아니라 가린다 — 다른 플랫폼에서 터질 때까지.**

- **#19 (허용오차가 시계의 눈금 크기를 빼먹었다)**: 위를 고친 뒤에도 5회 중 2회 실패했다.
  실패 폭이 **15ms, 16ms** — 값이 특정 숫자에 몰린 것이 진단의 단서였다. Windows 기본
  시스템 타이머 해상도가 **15.625ms** 이고 `GetTickCount64` 도 last-input 시각도 그 눈금
  단위로만 움직인다. 두 리더가 마이크로초 차이로 읽어도 그 사이에 눈금이 넘어가면 값이
  통째로 한 칸 벌어진다. 경과 시간만으로 허용오차를 세운 것이 원인.
  → 허용오차 = **측정한 경과 시간**(스케줄링 지연을 자기 보정) + **플랫폼 시계의 눈금
  크기**(물리적 성질). 눈금 값을 넣은 뒤 8연속 통과. **다른 클럭에서 온 두 값을 비교할 때는
  경과 시간뿐 아니라 그 클럭의 해상도도 허용오차에 들어가야 한다.** macOS 후보 비교(#22)도
  같은 함정 — CoreGraphics 는 초 단위 실수, IOKit 은 나노초라 눈금이 서로 다르다.

- **#21/#22 (게이트를 느슨하게 해서 산 초록불은 검증이 아니다)**: macOS 바인딩 후보 둘을
  CI 에서 한 배치로 비교했더니 CoreGraphics 가 IOKit 보다 **계통적으로 ~9ms 크게** 나왔다.
  결정적 근거는 **배치 소요가 0ms 인 표본**이었다 — 경과 시간으로 설명 안 되니 읽기 순서가
  아니라 진짜 오프셋. 허용오차를 조금만 넓혔으면 통과했고, 그랬으면 **이 비교가 존재한
  이유인 신호를 지우는** 셈이었다. 후보는 코드가 훨씬 단순하고(해제할 자원 0) 샌드박스
  검증도 통과했지만, 이 작업의 유일한 제약("기능 완전 동일")에서 탈락했다.
  → 후보 코드는 **머지하지 않고 닫았다.** 산출물은 코드가 아니라 **측정값**이었고, 그건
  이슈·PR·ADR-0004 에 남겼다. 지웠다 되돌릴 코드를 머지하려고 게이트를 일시적으로
  약화시키는 것보다 낫다. **측정하려고 만든 것은 측정되면 역할이 끝난다.**

## Step 5 — 테스트 신뢰 (커버리지가 두 번 거짓말한다)

- **#19 (커버리지에서 빼면 규칙도 같이 사라진다)**: `GetLastInputInfo` 실패 시 **0(활동
  중)을 반환** 하는 규칙이 FFI 호출 사이에 끼어 있어서, CI 호스트에서 실행 불가 →
  `// coverage:ignore` 대상이었다. 숫자는 맞지만 **완료 조건에 적힌 규칙이 미검증**으로
  남는다. 실패 여부를 `succeeded` 플래그로 순수 함수 안에 끌어들이니 테스트 가능한 규칙이
  됐고, 남은 ignore 구간은 결정이 없는 배관뿐이 됐다.
  → **`coverage:ignore` 를 붙이기 전에, 그 안에 결정(분기)이 들어 있는지 본다.
  결정이 있으면 ignore 가 아니라 순수 함수로 빼내는 게 답이다.**

- **#17 (라인 커버리지가 분기 갭을 가린다)**: `remaining()` 의 await 뒤 not-monitoring
  가드는 **줄로는 이미 커버돼** 있었다 — 그 줄은 실행됐지만 `Duration.zero` 를 돌려주는
  분기는 한 번도 안 탔다. 읽기를 park 시킨 뒤 stop 을 끼운 테스트를 넣어서야 그 분기가 탔다.
- **#16 (미커버 = 주입 seam 의 지도)**: `lib/` 미커버 3 줄은 전부 *테스트가 주입으로 대체한
  기본값의 본문* — `init()` 의 빈 콜백 둘과 기본 `Stopwatch` 클럭 람다. 미커버 목록은 버그가
  아니라 주입 seam 이 어디까지 뚫렸는지의 지도로 읽는다.
- **네이티브는 Dart 커버리지 분모에 없다.** `flutter test --coverage` 는 `lib/` 만 본다 —
  Dart 100% 여도 `windows/` C++·`macos/` Swift 는 0 줄 검증일 수 있다.

## Step 6/7 — 정합성 & 게이트

- **#19 (게이트가 안 밟던 경로에 잠복 결함이 있었다)**: Windows 잡에 `flutter test
  integration_test -d windows` 를 추가하자 CI 가 깨졌다 — 통합 테스트가 아니라 그 앞의
  CMake 생성에서. `windows/CMakeLists.txt` 의 gtest 블록이 googletest **1.11.0**
  (`cmake_minimum_required(2.8.12)`)을 pin 하고 있었고, **CMake 4 는 3.5 미만 호환을
  경고가 아니라 하드 에러로 제거**했다. 이 블록은 *example 앱을 빌드할 때만* 켜지는데
  (그 변수를 example 이 세팅한다), CI 가 example 의 Windows 앱을 빌드한 적이 한 번도
  없어서 지금까지 안 보였다. 로컬 CMake 는 4.0 미만이라 재현도 안 됐다.
  → 독립 하네스(`windows/test/`)가 쓰던 **1.12.1 로 통일**해서 해결. 판단 근거는 추측이
  아니라 **같은 CI 실행 안에 있었다** — 동일 러너에서 1.12.1 은 설정 성공(경고만),
  1.11.0 은 실패. **CI 에 새 단계를 추가하는 것은 새 코드를 검증하는 일이자, 그 단계가
  처음 밟는 경로를 검증하는 일이기도 하다.**

- **#10 (새 API 는 example 에서 쓰이는지 본다)**: `NotifyBefore` 는 구현·테스트가 끝난 뒤에도
  example 에 없었고, 별도 이슈로 채웠다.
- **`db4a8be` (`.pubignore` 가 `.gitignore` 를 무력화)**: 다른 repo 에서 복붙된
  `.pubignore` 가 존재하지도 않는 `CODE.md` 를 나열해, `.gitignore` 의 `build/` 가 아카이브에
  실렸다 — 테스트 하네스가 만든 **42 MB `flutter_windows.dll`** 포함 **64 MB** 패키지. 고친
  뒤 **216 KB**. `docs/` 를 빼면서 README 의 ADR 링크도 GitHub 절대경로로 돌려야 했다.
  **아카이브가 수 MB 를 넘으면 뭔가 새고 있는 것.**
